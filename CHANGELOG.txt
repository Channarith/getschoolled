- 2026-06-23 - Drive Mode audio: less robotic, correct accents. Web and mobile
  Drive Mode previously used the default Web Speech / expo-speech voice with no
  language set, so non-English content was mispronounced and English sounded
  robotic. Added voice-selection helpers (apps/web/app/lib/tts.ts,
  apps/mobile/src/tts.ts): map the UI locale to a BCP-47 tag, pick the device's
  best *natural/neural* voice for that language (Google/Microsoft "Natural"/
  "Online", Apple "Enhanced"/Siri, expo Enhanced quality), and use warmer
  prosody (pitch ~0.95). Course narration AND the Sala assistant now use it, and
  web voice recognition listens in the selected language. Unit-tested the voice
  ranking (prefers Natural over the legacy voice; picks the right language).
  Note: true human-grade audio needs a neural TTS backend (XTTS/cloud via the
  speech service) - proposed as the follow-up; this maximizes on-device quality.

- 2026-06-23 - Fix "can't create an account" (signup/login 404) on deployed
  single-origin hosts (bare IP / one domain). Root cause: the web bundle resolved
  every backend to its own page origin and then called bare paths (e.g.
  /auth/signup), which nothing routed -> 404. The NEXT_PUBLIC_*_URL contract was
  build-time-only and pointed at absolute api.* hosts that don't exist for a bare
  IP. Fix: apps/web/app/lib/api.ts now resolves backend bases in deployed mode to
  SAME-ORIGIN relative path prefixes (/identity, /curriculum, /orchestrator, ...)
  so calls hit the gateway on whatever host/IP/domain served the page - no DNS or
  build args needed. Added a single-origin edge gateway for docker-compose
  (infra/compose/edge.conf + an `edge` nginx service on port 80) that serves the
  web app at "/" and rewrites /<svc>/... to each service, and a hostless
  single-origin Ingress for Kubernetes (infra/k8s/ingress-origin.yaml, inherited
  by the VKE overlay) so the LB IP works without app/api subdomains. Updated the
  configmaps' NEXT_PUBLIC_*_URL to the relative prefixes. Verified end-to-end:
  signup now POSTs /identity/auth/signup through the gateway and returns 200,
  logging the user in.
- 2026-06-23 - Careers: connect to REAL job sites instead of a static board.
  aoep_shared.jobs gains live providers behind the existing JobsProvider
  interface: RemotiveJobsProvider + ArbeitnowJobsProvider (free, no key),
  AdzunaJobsProvider and JSearchJobsProvider (keyed aggregators that index
  LinkedIn/Indeed/Glassdoor/ZipRecruiter), and a CompositeJobsProvider that
  merges + de-dups. Each live posting is normalized to JobPosting with the real
  apply URL, the originating source label (e.g. linkedin/indeed), and skills
  derived from the title/description so course-matching still works; results are
  TTL-cached and resolvable by /jobs/{id}. Every live path falls back to the
  curated board on any network/parse failure, so /jobs never breaks offline.
  Selection is by env (dual-mode): default = curated board; JOBS_LIVE=1 enables
  the free boards; RAPIDAPI_KEY (JSearch) or ADZUNA_APP_ID/ADZUNA_APP_KEY enable
  LinkedIn/Indeed-sourced listings and take precedence; JOBS_PROVIDER pins one.
  Enabled JOBS_LIVE=1 in cloud configs (config/cloud.env, k8s + VKE configmaps,
  compose curriculum). Web /jobs now shows the live source per card and adds
  "View / Apply" links that open the original posting. Tests: real-shaped payload
  parsing (incl. a LinkedIn-sourced JSearch row), offline fallback, /jobs/{id}
  caching, and env-based provider selection. Note: real LinkedIn has no open API,
  so LinkedIn-sourced listings come via the JSearch/Adzuna aggregators or a
  LinkedIn partner key; live fetching needs cluster egress (falls back otherwise).

- 2026-06-23 - The AI teacher can award reward points. When a learner asks a
  substantive, on-topic question that yields a grounded answer, the orchestrator
  (the agent) mints a short-lived HMAC-signed reward voucher (scope=reward,
  bounded points, per-session cap via AGENT_REWARD_POINTS / AGENT_REWARD_SESSION_CAP)
  and returns it on the answer. The web Live Class redeems it at the new identity
  endpoint POST /rewards/grant, which verifies the agent's signature (shared
  INTERNAL_TOKEN_KEY) + expiry + one-time nonce before crediting the learner -
  so the agent authorizes the points while a user cannot forge or replay them.
  The chat shows "🎉 The AI teacher awarded you N points". New: Answer.reward
  (orchestrator), AccountStore.award_grant + /rewards/grant (identity),
  grantReward() web helper. Tests: services/identity/tests/test_rewards_api.py
  (valid/forged/replay/over-cap/unconfigured) and
  services/orchestrator/tests/test_agent_reward.py (voucher minted, per-session
  cap, no key -> no reward, trivial question -> no reward). Requires
  INTERNAL_TOKEN_KEY shared by orchestrator + identity.
