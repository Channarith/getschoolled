AOEP - DESIGN & ARCHITECTURE
============================
Agentic Online Education Platform. This is the text source-of-truth for the
system design; GitHub-facing rendered diagrams live in the README as mermaid.
No generated diagram/image assets are currently checked in.

No-markdown house rule: this file is plain text on purpose. Diagrams here are
ASCII; the README keeps the rendered GitHub version.

-----------------------------------------------------------------------------
1. SYSTEM CONTEXT (who uses it)
-----------------------------------------------------------------------------
  Free learner / visitor ....... browse catalog, ad-supported watch, live class
  Paid member (Pro/Premium) .... ad-free watch, full catalog, rewards
  Parent / account holder ...... many learner sub-profiles, mastery, portfolio
  Student sub-profile .......... the person actually taking a class
  Educator / institution ....... Human-in-the-Loop review (co-teach / co-grade)
  Platform admin / ops ......... feature flags, versions, data-mining insights
  Partner / integrator ......... LMS/LTI, finance, cloud connectors, webhooks

4. KEY DATA FLOWS
-----------------------------------------------------------------------------
4a. Ask the AI teacher (grounded, optionally human-reviewed)
    web -> orchestrator /api/sessions/{id}/ask
         -> RAG retrieve (curriculum/sample-curriculum)
         -> Tutor compose answer grounded in passages
         -> groundedness guard (flag unsupported claims, confidence)
         -> HIL: should_escalate(confidence,risk,autonomy)?
               yes -> ReviewQueue (educator /console approves/edits) -> student
               no  -> answer returned with citations + confidence

4b. Post-class survey -> multi-dimensional data mart (data mining)
    admin enables engagement.post_class_survey (flag, memory)
    web /class "Finish" -> GET /survey/post-class (gated) -> modal
    submit -> POST /survey/post-class -> SurveyStore
    admin GET /admin/survey/insights -> OLAP cube (course x class_type x rating)
          + mined free-text suggestion themes

4c. Ad decisioning (monetization)
    web /watch -> curriculum GET /courses/{id}/ad-breaks?tier=
         tier in {pro,premium} -> ad-free (empty plan)
         else -> pre-roll + capped mid-rolls (VAST/VMAP), skippable

4d. Rewards (points -> prizes)
    identity set_status(passed) -> PointsLedger += points_for_completion
    GET /rewards -> balance + ledger; POST /rewards/redeem -> voucher / raffle

4e. Personalization (Foresight)
    curriculum POST /recommend(student profile) -> Foresight engine
         (attention, multimodal fuse, router, pattern attention, multi-head)
         -> ranked recommendations + skill gaps + adapted difficulty

5. ADMIN / OPS SURFACE
-----------------------------------------------------------------------------
  Feature flags (memory): public GET /flags/evaluate, /flags/{key};
    admin (X-Admin-Secret == ADMIN_SECRET) PUT/override/reset.
    Catalog ~30 flags across engagement/data/access/monetization/ai/ux/ops.
  Version visibility: every service GET /version (version, git_sha, build_time,
    api_version, deploy_mode); GET /__meta (route index for automation).
    Web /admin "System & Versions" panel aggregates + flags version drift.
  Automation hooks (gated by ADMIN_SECRET AND ENABLE_TEST_ENDPOINTS):
    POST /admin/test/reset|seed for deterministic E2E setup.
  Observability (every service, dependency-free; local + cloud):
    GET /metrics            Prometheus exposition (perf, memory, requests, latency)
    GET /telemetry/summary  perf + memory/runtime (RSS/CPU/GC/threads) + counts
    GET /telemetry/errors   recent exceptions w/ traceback + request id (RCA)
    GET /telemetry/logs     recent structured events
    Request middleware stamps X-Request-ID + Server-Timing and records per-route
    latency/error stats; optional Sentry/OTLP export via env when libs present.
    Web /admin "Observability & Telemetry" panel aggregates all services.
  QA: qa/stress.py (all 8 services; health/version/__meta + endpoint scenarios),
    qa/regression.py (pytest+coverage, web typecheck/lint, stress smoke).

-----------------------------------------------------------------------------
6. DEPLOYMENT TOPOLOGY (dual/triple mode)
-----------------------------------------------------------------------------
  DEPLOY_MODE=local  docker compose; mocks/sandboxes; everything offline-runnable
  DEPLOY_MODE=cloud  Kubernetes + GPU pool; real LLM/speech/vision; Stripe; S3
  DEPLOY_MODE=edge   on-device/humanoid image (apps/agent-runtime/edge); offline
  Each capability (llm/speech/vision/media/object_store/payment/ocr/...) can be
  overridden independently by <COMPONENT>_MODE without forking code, e.g. keep
  biometrics local even in cloud for compliance.
