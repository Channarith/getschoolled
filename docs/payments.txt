3. ROUTING IN CLOUD MODE (RoutedPaymentProvider)
----------------------------------------------------------------------------
When DEPLOY_MODE=cloud the factory returns a RoutedPaymentProvider
that fans method-specific checkouts out to the right regional
processor. The set of methods advertised in /payment-methods is the
union of every CONFIGURED provider's method set (a provider is
configured iff its API key is non-empty). Unconfigured methods raise
NotImplementedError on /checkout — fail-closed.

Local mode (DEPLOY_MODE=local) returns the SandboxPaymentProvider
which simulates every method so the entire UX (UPI, PIX, KHQR, etc.)
is testable offline without any cloud credentials.

----------------------------------------------------------------------------
4. COUNTRY → METHOD ORDERING
----------------------------------------------------------------------------
COUNTRY_METHODS in payments.py maps ISO-3166-1 alpha-2 codes to an
ordered list of methods popular in that country. The /payment-methods
API exposes this:

  GET /payment-methods                  -> all 44 methods
  GET /payment-methods?country=US       -> ordered list for US
  GET /payment-methods?country=BR       -> ordered list for Brazil
  GET /payment-methods?locale=km        -> infers country=KH (Cambodia)
  GET /payment-methods?locale=hi        -> infers country=IN (India)
  GET /payment-methods/by-country       -> full matrix (countries +
                                            locale->country defaults)

The web + mobile checkout UIs hit /payment-methods?locale=<active>
to render a market-appropriate picker without a separate config step.

6. WEBHOOKS
----------------------------------------------------------------------------
Inbound payment webhooks are received at:
  POST /payments/webhook/{provider}   on the integrations service
  POST /webhooks/inbound/{provider}   for generic provider payloads

Each is HMAC-SHA256 verified against
  {PROVIDER}_WEBHOOK_SECRET            preferred; per-provider secret
  WEBHOOK_SIGNING_KEY                  platform-wide fallback
  "dev-webhook-secret"                 LOCAL deploy mode only

In cloud deploy mode an unset key fails CLOSED — the legacy dev
secret is NOT used as a fallback.

The handler normalises Stripe-style events via parse_payment_event()
in connectors/finance.py; one path covers Stripe + Mercado Pago +
PayPal-style payloads. Provider-specific normalisation can be added
to that file when a processor uses a non-Stripe-shaped event.
