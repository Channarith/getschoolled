SALAREEN / AOEP — API & WEBHOOK REFERENCE
=========================================
Plain text on purpose (repo no-markdown house rule; see AGENTS.md).

This is the canonical reference for every HTTP API and webhook the platform
exposes. It is generated from and kept in lockstep with the service source
(services/<name>/src/<pkg>/main.py) and the shared library
(packages/shared/src/aoep_shared/*). Companion docs:

  docs/api-use-cases.txt          - end-to-end recipes (curl) for real flows
  docs/api-cost-value.txt         - cost-to-value analysis per API/provider
  docs/integrations-jobs-careers.txt - how Careers connects to job sites
  docs/payments-and-security.txt  - transactions + how they are secured
  docs/payments.txt               - the 44 supported payment methods
  docs/secrets.txt                - every secret/env var and where it is used

============================================================================
0. SERVICES AT A GLANCE
============================================================================
Backend = Python/FastAPI microservices behind a provider abstraction
(packages/shared). Each service listens on container port 8000; locally they
are mapped to distinct host ports (infra/compose/docker-compose.yml). In the
cluster they are reached service-to-service by name (http://<svc>:8000) and
from the browser through the ingress (infra/k8s/ingress.yaml).

  Service        Local URL                Package           Owns
  -----------    ----------------------   ---------------   -------------------------
  orchestrator   http://localhost:8000    orchestrator      live class loop, director, assessment, HIL, group classes
  speech         http://localhost:8002    speech_gw         TTS/ASR planning, translation, language learning
  perception     http://localhost:8003    perception        face detect/identify (on-device + server), consent gate
  memory         http://localhost:8004    memory            consent, legal notices, retention, feature flags, surveys
  curriculum     http://localhost:8005    curriculum        catalog, decks, RAG, jobs/careers, homework, audio courses
  billing        http://localhost:8006    billing            plans, entitlement checks, payment methods, checkout
  integrations   http://localhost:8007    integrations      webhooks, payment webhooks, LMS/SIS, calendar, SSO, media bridges
  identity       http://localhost:8008    identity          auth, students, enrollments, rewards, games, portfolio

The Next.js web app (apps/web) is the browser client; it resolves each service
base URL from NEXT_PUBLIC_*_URL or the page origin (apps/web/app/lib/api.ts).

1. CONVENTIONS (apply to EVERY service)
============================================================================
Every service is built by aoep_shared.service.create_service(), which adds a
uniform contract:

1.1 Standard endpoints (present on all services)
  GET  /health        -> {service, status:"ok", deploy_mode, components, version}
  GET  /version       -> {service, deploy_mode, git_sha, built_at, ...}
  GET  /metrics       -> Prometheus exposition (text/plain)
  GET  /telemetry/summary | /telemetry/errors | /telemetry/logs -> in-proc telemetry
  GET  /openapi.json, /docs (Swagger UI), /redoc -> live, per-service API schema
  ==> The single most authoritative spec is the live /openapi.json of each
      service; this document is the human-readable companion to it.

1.2 Auth model (three tiers)
  a) PUBLIC            - most read + learner-facing endpoints. No auth.
  b) USER (bearer)     - identity routes use a session JWT in
                         Authorization: Bearer <token> (see /auth/login). The
                         web client stores it in localStorage (aoep_token).
  c) INTERNAL-ONLY     - agent/teacher tools (homework grader, corrections
                         bulk apply, payouts, retention purge, catalog export,
                         webhook emit, admin flag writes). Gated by
                         Depends(require_internal): supply header
                         X-Internal-Token: <token>. Fail-CLOSED: if no internal
                         key is configured the gate DENIES (never ships open).
                         Token is either a static INTERNAL_TOKEN (dev) or an
                         HMAC-signed token (INTERNAL_TOKEN_KEY) with scope in
                         {internal, agent, teacher}. Bypass for local tests:
                         INTERNAL_AUTH_DISABLED=1.
     Source: packages/shared/src/aoep_shared/internal_auth.py

1.3 Rate limiting
  Per-principal token bucket (default 120 req / 60s; tune with RATE_LIMIT,
  RATE_LIMIT_WINDOW). Key = X-User-Id, else X-Forwarded-For, else client IP.
  Over limit -> HTTP 429 with Retry-After + X-RateLimit-Limit/Remaining headers.
  Health/metrics/docs are never throttled. Disable: RATE_LIMIT_DISABLED=1.
  The nginx ingress adds a second, edge-level limit (limit-rps).

1.4 Caching, tracing, CORS
  - Hot GETs send Cache-Control + ETag (conditional GET -> 304). Disable:
    HTTP_CACHE_DISABLED=1.
  - Every response carries X-Request-ID (echoes inbound x-request-id) and
    Server-Timing: app;dur=<ms>.
  - CORS allow-list via CORS_ORIGINS (comma-separated; default "*" for local).

1.5 Errors
  Standard FastAPI JSON: {"detail": "..."} with conventional status codes:
  400 bad input, 401 bad webhook signature, 403 internal-only,
  404 unknown id, 409 conflict (e.g. class full), 422 validation,
  429 rate limited, 503 real provider unavailable (offline/no key).

6. PERCEPTION (machine vision, opt-in + on-device)  -  :8003
============================================================================
Real, self-hosted OpenCV YuNet (detect) + SFace (embed) on CPU. The HYBRID
path runs detect+embed in the browser and sends only the 128-d embedding.

  POST /enroll/{student_id}            enroll from an image (server-side)
  POST /identify                       identify faces in an image (server-side)
  POST /enroll-embedding/{student_id}  enroll from a precomputed embedding (hybrid)
  POST /identify-embedding             match precomputed embeddings (hybrid)
  GET  /vision/models/{name}           serve ONNX model weights to the browser
  POST /analyze/consent-check          compliance/consent gate before matching
  GET  /gallery                        enrolled gallery summary
  Privacy: identification is gated on per-student consent (memory service) and
  the EU-AI-Act-aware policy engine; raw frames never leave the device on the
  hybrid path.

10. WEBHOOKS — SIGNING & VERIFICATION  (the security contract)
============================================================================
Source of truth: packages/shared/src/aoep_shared/webhooks.py and the
integrations service handlers.

10.1 Outbound (we -> subscriber)
  - Register a subscriber via POST /webhooks/subscriptions with an optional
    per-subscriber `secret`.
  - On each event we POST the canonical JSON body and set header
        X-AOEP-Signature: hex( HMAC_SHA256(secret, body_bytes) )
    body_bytes = json.dumps(event, sort_keys=True, separators=(",",":")).
  - Delivery retries with backoff; results are reported by /webhooks/emit.
  - Subscriber MUST recompute the HMAC over the RAW body and compare in
    constant time (hmac.compare_digest) before trusting the payload.

10.2 Inbound (provider -> us)  POST /webhooks/inbound/{provider},
                               POST /payments/webhook/{provider}
  - We read the RAW request body and the header X-AOEP-Signature (or
    X-Signature) and verify HMAC_SHA256 against the provider secret.
  - Secret resolution order (fail-closed in cloud):
        1. {PROVIDER}_WEBHOOK_SECRET   (per-provider override)
        2. WEBHOOK_SIGNING_KEY         (platform default)
        3. "dev-webhook-secret"        (DEPLOY_MODE=local ONLY)
    In cloud, if neither env is set the secret is "" so verification FAILS
    (401) rather than accepting payloads signed with a known dev key.
  - Invalid/missing signature -> HTTP 401 "invalid webhook signature".

10.3 Event types emitted (examples)
  enrollment.paid           (after a verified payment grant)
  ...register a subscriber with empty event_types to receive ALL events.

11. EVOLVING THIS DOC
============================================================================
When you add or change a route, update this file AND verify it stays in sync:
  python3 scripts/check_api_docs.py
That script parses every services/*/main.py for route decorators and asserts
each path is mentioned here (and flags documented paths that no longer exist).
Run it in CI alongside the service tests. The per-service /openapi.json remains
the machine-readable source of truth.
