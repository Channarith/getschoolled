SALAREEN - INFRA BOOTSTRAP RUNBOOK
==================================
Plain text; pair with docs/hosting.txt.

This is the step-by-step that turns "we picked AWS + Cloudflare" into a
running cluster. The Terraform under infra/terraform/{aws,cloudflare}
is intentionally a *skeleton*: no upstream community modules pinned to
specific versions, no production-bake-in. Read every file before you
apply. Plan-then-apply, never apply blind.

PREREQUISITES
-------------
  - Terraform >= 1.6
  - AWS CLI v2; `aws sts get-caller-identity` returns the staging account
  - kubectl >= 1.29; `kubectl version --client`
  - awscli plugins: `aws eks update-kubeconfig`
  - Cloudflare account; an API Token with Zone:Read + Zone:Edit on
    salareen.com and Account:Read

ACCOUNTS
--------
  prod   -> 1234567890XX
  stg    -> 0987654321XX
  shared -> 1122334455XX
  Use IAM Identity Center to log into them; no static keys.

REGIONS
-------
  Primary: us-east-1
  DR:      us-west-2 (later, Phase 2)

ORDER OF OPERATIONS
-------------------
  1. Cloudflare zone (DNS-only at first, no proxy yet).
  2. AWS networking (VPC + subnets + NAT + endpoints).
  3. AWS data plane (RDS Aurora + ElastiCache + S3 buckets).
  4. AWS compute plane (EKS cluster + node groups).
  5. Cluster bootstrap (cluster autoscaler, ALB controller,
     external-DNS, cert-manager, kube-prometheus-stack).
  6. Application deploy (kustomize build infra/k8s | kubectl apply).
  7. Cloudflare proxy ON + WAF rules + cache rules.
  8. Smoke test (qa/loadtest.py).

STEP 1 - CLOUDFLARE ZONE
------------------------
  cd infra/terraform/cloudflare
  cp terraform.tfvars.example terraform.tfvars
  # Fill in: cloudflare_api_token, account_id, domain
  terraform init
  terraform plan
  terraform apply
  # Update the registrar's nameservers to the ones Cloudflare prints.
  # DNSSEC: enable at the registrar AFTER Cloudflare reports zone is
  # active.

STEP 2 - AWS BASE
-----------------
  cd infra/terraform/aws
  cp terraform.tfvars.example terraform.tfvars
  # Fill in: project, region, az_count, db_password (better: AWS
  # Secrets Manager bootstrap; never check in plain).
  terraform init -backend-config="bucket=salareen-tfstate-PROD" \
                 -backend-config="key=aws/main.tfstate" \
                 -backend-config="region=us-east-1"
  terraform plan
  terraform apply
  # ~25 min: VPC, subnets, NAT, EKS, node groups, RDS, Redis, S3.

STEP 3 - kubectl ACCESS
-----------------------
  aws eks update-kubeconfig --name salareen-prod --region us-east-1
  kubectl get nodes        # expect 6 nodes across 3 AZs
  kubectl get pods -A      # the EKS add-ons should be running

STEP 4 - CLUSTER BOOTSTRAP (helm; not in TF on purpose - simpler to roll)
------------------------------------------------------------------------
  helm repo add eks https://aws.github.io/eks-charts
  helm repo add jetstack https://charts.jetstack.io
  helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
  helm repo add external-dns https://kubernetes-sigs.github.io/external-dns
  helm repo update

  helm upgrade -i aws-lbc       eks/aws-load-balancer-controller \
       -n kube-system --set clusterName=salareen-prod \
       --set serviceAccount.create=true
  helm upgrade -i cert-manager  jetstack/cert-manager \
       -n cert-manager --create-namespace --set installCRDs=true
  helm upgrade -i external-dns  external-dns/external-dns \
       -n external-dns --create-namespace \
       --set provider=cloudflare --set cloudflare.proxied=true
  helm upgrade -i kube-prom     prometheus-community/kube-prometheus-stack \
       -n monitoring --create-namespace

STEP 5 - APP MANIFESTS
----------------------
  kubectl create namespace aoep
  # Provide the secrets the manifests reference (DATABASE_URL, etc.):
  kubectl -n aoep apply -f - <<EOF
  apiVersion: v1
  kind: Secret
  metadata: { name: aoep-secrets }
  stringData:
    DATABASE_URL: "postgres://..."
    LIVEKIT_API_KEY: "..."
    LIVEKIT_API_SECRET: "..."
    OBJECT_STORE_ACCESS_KEY: "..."
    OBJECT_STORE_SECRET_KEY: "..."
    PAYMENT_API_KEY: "..."
  EOF
  # Better: External Secrets Operator + AWS Secrets Manager.

  cd /workspace
  kustomize build infra/k8s | kubectl apply -f -
  kubectl -n aoep rollout status deploy/curriculum
  kubectl -n aoep rollout status deploy/orchestrator

STEP 6 - CLOUDFLARE PROXY ON
----------------------------
  In infra/terraform/cloudflare/main.tf, set proxied = true on the
  api/app records, run plan + apply. Cloudflare auto-issues the edge
  cert. Origin cert from ACM (created by Terraform in step 2) is
  attached to the ALB.

STEP 7 - SMOKE TEST
-------------------
  python3 qa/loadtest.py https://api.salareen.com/audio/categories \
      --rps 500 --duration 30
  # Expect: p95 < 100ms, error_ratio < 0.5%, cache_hit_ratio rising
  # over the 30s as Cloudflare warms.

DESTROY (staging)
-----------------
  cd infra/terraform/aws
  terraform destroy
  cd ../cloudflare
  terraform destroy

STATE STORAGE
-------------
  Each environment uses an S3 backend with DynamoDB for state lock.
  Create those by hand once, in a SEPARATE bootstrap account, and
  never inside the same Terraform that uses them.

      aws s3api create-bucket --bucket salareen-tfstate-PROD \
          --region us-east-1
      aws s3api put-bucket-versioning --bucket salareen-tfstate-PROD \
          --versioning-configuration Status=Enabled
      aws dynamodb create-table --table-name salareen-tfstate-locks \
          --attribute-definitions AttributeName=LockID,AttributeType=S \
          --key-schema AttributeName=LockID,KeyType=HASH \
          --billing-mode PAY_PER_REQUEST --region us-east-1

NEVER CHECK IN
--------------
  *.tfstate, *.tfstate.backup, *.tfvars (use *.tfvars.example),
  any kubeconfig with embedded secrets, any AWS access key, any
  Cloudflare token.
